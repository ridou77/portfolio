Dolibarr v18 → v22 : Évolutions Majeures et Impact sur les Modules Personnalisés

Analyse des changements structurels introduits sur quatre versions majeures de l'ERP open-source Dolibarr. Quelles nouvelles API, quels hooks et quelles ruptures de compatibilité peuvent affecter vos modules custom ?

Contexte

Depuis mon arrivée en alternance chez Emdelen en septembre 2024, j'ai été immédiatement confronté à la nécessité de maintenir des modules PHP compatibles avec les dernières versions de Dolibarr. La veille sur les releases notes GitHub est donc devenue une routine hebdomadaire.

Principales évolutions v18 → v22

  • Système de Hooks renforcé : introduction de nouveaux points d'accroche dans les objets métier (Commande, Facture, Projet), permettant d'éviter les patches directs du core.
  • API REST enrichie : de nombreux endpoints ont été stabilisés et documentés, rendant les intégrations tierces plus robustes.
  • Réécriture partielle du moteur de droits : modifications dans la gestion des $user->rights, avec impact sur les modules qui surchargeaient les permissions.
  • Nouvelles Extrafields : possibilité d'attacher des champs additionnels à davantage d'objets, avec support amélioré dans les formulaires générés automatiquement.
  • Performance : optimisation des requêtes SQL dans les listes longues, réduction des N+1 queries dans les agendas et projets.
💡 Conseil pratique : avant toute mise à jour majeure de Dolibarr, toujours tester les modules custom sur un environnement de staging avec la nouvelle version, en activant le mode debug define('MAIN_CORE_DEBUG', 1).

Impact sur mes modules

Lors de la mise à jour vers v18 puis v22, j'ai dû réviser la gestion des hooks dans le Module Générique Emdelen pour utiliser les nouveaux points d'accroche sans modifier le core. J'ai également mis à jour les requêtes SQL pour m'aligner avec les changements de schéma des tables de projets.

DoliDroid : Accéder à Dolibarr depuis un Smartphone Android

Découverte de DoliDroid, l'application Android communautaire qui permet d'utiliser Dolibarr en mobilité. Analyse des fonctionnalités, limites et cas d'usage pertinents en contexte professionnel.

Présentation

DoliDroid est une application Android développée et maintenue par des membres de la communauté Dolibarr. Elle offre une interface mobile adaptée pour accéder aux principales fonctionnalités de l'ERP depuis un smartphone.

Fonctionnalités couvertes

  • Consultation et création de devis, factures, commandes
  • Accès au CRM (tiers, contacts)
  • Gestion des agendas et des événements
  • Pointage de temps sur des projets
  • Notifications push pour les tâches et alertes

Limites identifiées

Les modules personnalisés ne sont pas automatiquement accessibles depuis DoliDroid : l'application ne supporte que les modules officiels de Dolibarr. Les extrafields ajoutés via des modules custom ne s'affichent pas nativement.

🔍 Piste d'amélioration : utiliser l'API REST de Dolibarr pour exposer les données des modules custom, ce qui permettrait d'y accéder depuis une application mobile maison ou depuis des outils no-code comme n8n.

Pertinence pour Emdelen

La mobilité est un axe à explorer pour les chefs de projet qui suivent plusieurs chantiers en déplacement. La combinaison DoliDroid + API REST maison pourrait répondre à ce besoin à terme.

Hooks et Triggers Dolibarr : Architecture Propre sans Toucher au Core

Le système de hooks de Dolibarr permet d'étendre les comportements de l'ERP sans modifier son code source. Comprendre ce mécanisme est fondamental pour produire des modules robustes et pérennes.

Différence entre Hook et Trigger

Un Hook est un point d'injection dans l'affichage ou le traitement, appelé par Dolibarr à des moments clés du rendu. Un Trigger est déclenché lors d'actions métier (création, modification, suppression d'objets) et s'exécute après validation.

Créer un Hook

  • Créer la classe InterfaceMyHook dans /core/triggers/interface_mymodule.class.php
  • Implémenter la méthode runWorkflow() ou executeHooks()
  • Déclarer le hook dans le module descriptor
💡 Bonne pratique : toujours vérifier l'action courante avec $action et le contexte avec $hookmanager->contextarray avant d'agir, pour éviter des effets de bord indésirables.

Application dans mes modules

Dans le Module Générique Emdelen, j'utilise des hooks pour injecter des champs supplémentaires dans les fiches fournisseurs et dans les formulaires de projets, sans modifier les templates natifs de Dolibarr.

Extrafields Dolibarr : Étendre les Objets sans Modifier le Schéma Core

Les extrafields permettent d'ajouter des attributs personnalisés à n'importe quel objet Dolibarr (tiers, projet, facture…) sans toucher aux tables natives. Un outil puissant — à condition de bien en connaître les limites.

Fonctionnement

Les extrafields sont stockées dans des tables suffixées _extrafields (ex: llx_societe_extrafields). Elles sont déclarées soit via l'interface administrateur de Dolibarr, soit directement dans le code du module avec la classe ExtraFields.

Types disponibles

  • varchar, int, double, date : types simples
  • select : liste déroulante avec valeurs prédéfinies
  • link : lien vers un autre objet Dolibarr (ex: un projet)
  • checkbox, radio : pour des options binaires ou multiples
  • text, html : champs libres, avec éditeur WYSIWYG pour html
⚡ Astuce : déclarer les extrafields dans le code du module plutôt que via l'UI permet de les versionner avec Git et de les déployer automatiquement à l'installation du module.

Utilisation dans mes modules

J'utilise les extrafields pour stocker des données spécifiques à Emdelen sur les tiers fournisseurs (notation, catégorie interne) et sur les projets (charge prévisionnelle, référence client), sans jamais modifier les tables core de Dolibarr.
